## 기획 단계에서 느낀 점
- 일단 기획부터 병렬로 하고 싶어서 아래대로 진행. 사실상 병렬...?은 아니었음.

> 프로젝트 전체 기획자
 너는 프로젝트 전체 기획자야. 현재 프로젝트 전체 파일에 기본 기획 내용이 있어. 
 기본 기획 내용 plan.md, supabase.md에서 피드백을 받았어(feedback.md). 이제   
  너는 피드백을 기반으로 더 나은 결과를 내야 돼. 피드백을 완전히 수용하지       
  않아도 되고, 필요한 것만 수용해서 최종 기획안을 작성해. 그 기획안에는 기술    
  트리를 모두 포함하고 있어야 해. 기본 규칙은 CLAUDE.md를 따라.  

> 사용자
 너는 oriverse를 이용하는 사용자야. final-plan.md가 실행되었을 때, 사용자      
  입장에서 불편할 만한 점이 있을지 정리해서 feedback2.md 파일 만들어. 

> 백엔드 QA
너는 백엔드 QA 전문가야. final-plan.md가 실행되었을 때 발생할 수 있는 문제를  
  테스트하고 있어. feedback_frombackqa.md 파일을 만들어서 발생한 모든 문제를    
  작성해.   

> 프론트 QA
너는 프론트 QA 전문가야. final-plan.md가 실행되었을 때 발생할 수 있는 프론트  
  테스트 문제를 작성하고 있어. feedback_fromfrontqa.md 파일을 만들어서 문제를   
  정리해. 사용자 경험에 집중하고, 구현 가능성에도 집중해. 

> 소비자 심리 전문가
  너는 소비자 심리 전문가야. final-plan.md를 읽고 oriverse의 사용자의 총 결제       
  금액이 제일 많이 나올 수 있게 수 있게 feedback_frompsy.md 파일을 만들어서        
  정리해. 지금 있는 결제 시스템에서 엄청 크게 바꾸지 말고, 예를 들어 어느           
  시점에서 결제하게 만들어야 소비자가 심리적으로 가장 결제할 확률이 높은지를        
  비즈니스 심리학적으로, 실제 케이스를 기반으로 고려해서 피드백 파일을 작성해.   

- 근데 원래 병렬 처리에서 가장 효과를 볼 수 있는 첫 번째 방법이 task별로 나눠서 할당하는 건데, 기획은 그게 안 돼서 문제.
- 그리고 위 agents는 사실 기획에서 일회성으로만 사용하는 agents라서 토큰 잡아먹는 거 생각했을 때 굳이 싶기도 함.
- 그냥 얌전히 일단 처음 해봤다고 생각하고 코드 짤 때 task 주고 서로 피드백하는 agents로 다시 써봐야겠음.
